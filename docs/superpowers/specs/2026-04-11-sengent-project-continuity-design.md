# Sengent Project Continuity Design

## Goal

为 `Sengent` 建立一套面向非工程背景维护者的长期推进机制，让维护者在离开项目几天后重新回来时，仍然可以在 2 分钟内重新进入状态，并知道唯一推荐的下一步。

这次设计解决的不是“自动化不够多”，而是“项目上下文没有稳定入口”。核心目标是降低重新上手成本，而不是增加新的工程系统。

## Problem

当前项目已经有较完整的代码、文档和历史设计材料，但这对长期推进并不天然等于“容易继续做下去”。

对于当前维护方式，真正的问题主要是：

- 过几天回来后，不容易立刻知道项目现在处于什么阶段
- 即使能看懂最近做过什么，也不容易知道下一步该做什么
- 如果把长期记忆留在聊天线程里，线程会越来越长，重新进入项目的成本会越来越高
- 如果同时开很多执行线程，又容易失去“哪条线程代表最新状态”的控制感

所以这次设计不把聊天线程当作长期记忆系统，而把线程降级成执行通道，把文档提升为项目记忆的主载体。

## Design Principles

这套机制遵守四条原则：

1. 驾驶舱优先，不靠回忆。
   重新进入项目时，优先读取一页固定文档，而不是先翻聊天历史、代码或 commit。

2. 主线唯一，不并行发散。
   项目在任何时刻都应该只有一个被明确推荐的下一步。

3. 执行与记忆分离。
   分线程负责完成具体任务，长期记忆由文档承担。

4. 先可控，再自动化。
   第一阶段采用半自动调度，不追求全自动派发分线程。

## Scope

本设计覆盖：

- 一个固定的项目驾驶舱文档
- 一个主线程加少量分线程的工作方式
- 分线程创建条件、交付格式和回写规则
- 每次开工和收尾时的最小更新流程

本设计不覆盖：

- 全自动线程编排系统
- 复杂的任务管理软件接入
- 自动读取所有 commit 生成日报
- 新的运行时代码路径改造

## Recommended Operating Model

推荐使用 `1 + N` 模型：

- 1 个主线程
- N 个短生命周期分线程

其中：

- 主线程负责项目主线判断，不负责承载所有细节执行
- 分线程只处理一个清晰的小任务
- 长期状态不保存在线程里，而保存在项目驾驶舱文档里

这比“一个线程走到底”更稳，也比“很多线程轮替但没有总账本”更容易维护。

## Main Thread Responsibilities

主线程只负责四类事情：

1. 读取并维护项目驾驶舱
2. 判断当前最重要的一个任务
3. 决定是否需要为该任务创建分线程
4. 在分线程结束后，把结果回写到驾驶舱

主线程不负责：

- 长期保存所有执行细节
- 同时推进多个同等级任务
- 代替分线程完成复杂调查或实现工作

换句话说，主线程是项目调度台，不是项目仓库本身。

## Task Thread Responsibilities

分线程只负责执行，不负责保存长期记忆。

每个分线程必须满足：

- 只处理一个明确任务
- 有清晰的完成条件
- 做完后可以被废弃

适合分线程处理的任务包括：

- 修一个明确的小功能
- 调查一个具体 bug 或异常行为
- 做一次容易跑偏的小实验
- 补一段文档或测试

不适合开分线程的情况包括：

- 只是想“随便看看”
- 任务目标还没有被主线程说清楚
- 同时混合实现、调查、重构三个方向

## Thread Orchestration

第一阶段采用半自动调度，不做完全自动派发。

推荐规则是：

- 主线程负责提出“是否该开分线程”
- 主线程负责给出一句清晰的任务描述
- 维护者负责确认并启动分线程
- 分线程完成后，由主线程吸收结果并更新驾驶舱

之所以不直接上全自动，是因为当前最重要的不是省一次点击，而是保持整个流程可理解、可控、可复盘。

## Project Dashboard

项目驾驶舱建议固定在：

- `docs/current-state.md`

这个文件是项目的唯一首页。维护者每次重新进入项目时，先看它，而不是先看代码或旧线程。

文件结构应固定为以下栏目：

- `项目阶段`
- `现在能工作的功能`
- `最近完成的 3 件事`
- `当前阻塞 / 风险`
- `唯一推荐的下一步`
- `如果今天只有 30 分钟，做什么`
- `最近一次分线程结果`

其中最重要的是两项：

- `当前阻塞 / 风险`
- `唯一推荐的下一步`

如果这两项不清楚，维护者就很容易重新陷入“知道很多，但不知道该做什么”。

## Dashboard Content Rules

驾驶舱不是长篇项目报告。它只保留高信号信息。

更新时应遵守这些规则：

- 最近完成事项最多保留 3 条
- 当前阻塞最多保留 3 条
- 唯一推荐下一步只能有 1 条
- 如果存在多个候选任务，主线程必须先帮维护者选出一个
- 语言尽量直接，避免抽象术语堆积

驾驶舱的目标是帮助重新上手，不是帮助追溯所有历史。

## Task Thread Handoff Contract

每个分线程结束时，必须交回固定格式的结果：

1. 做了什么
2. 如何验证
3. 还剩什么问题
4. 建议主线程下一步做什么

主线程接收后，要把这些信息压缩进：

- `最近完成的 3 件事`
- `当前阻塞 / 风险`
- `唯一推荐的下一步`
- `最近一次分线程结果`

分线程本身可以随后废弃，因为长期有价值的信息已经被主线程吸收到了文档里。

## Daily Workflow

### Start Of Work

每次开工，按以下顺序进行：

1. 先读 `docs/current-state.md`
2. 确认项目当前阶段和唯一推荐下一步
3. 判断这一步是否需要新开分线程
4. 如果需要，再启动一个短生命周期分线程

不推荐的开工方式：

- 直接翻代码
- 直接翻旧聊天记录
- 因为焦虑而同时开启多个方向

### End Of Work

每次收尾，按以下顺序进行：

1. 总结本次实际完成内容
2. 记录验证结果
3. 写清楚遗留问题
4. 更新 `docs/current-state.md`
5. 只保留一个明确的下一步

如果这一步没有完成，下一次重新进入项目时，维护者仍然会重新面对同一个问题。

## Concurrency Rule

为了控制复杂度，第一阶段建议：

- 同时活跃的分线程不超过 2 个

超过这个数量后，维护者会重新失去对“哪件事最重要”的把握。对于当前项目阶段，更高的并行度不会带来更高的推进效率，反而会增加恢复上下文的成本。

## Why This Fits The Current Repository

当前仓库已经有大量长期材料：

- `docs/project-context.md`
- `docs/sengent-maintainer-guide.md`
- `docs/superpowers/specs/`
- `docs/superpowers/plans/`

这些材料适合保存架构背景、维护流程和阶段设计，但还缺少一个“重新回来后第一眼该看什么”的首页。

因此最合适的补强不是再增加更多深文档，而是补一个轻量、高频更新的驾驶舱文档，并让主线程围绕它工作。

## Adoption Plan

第一阶段只需要落地三件事：

1. 新增 `docs/current-state.md`
2. 新增一份给维护者看的协作说明，定义主线程 / 分线程规则
3. 约定每次任务结束后都要更新驾驶舱

等这套机制稳定后，再考虑：

- 用 CLI 辅助生成驾驶舱摘要
- 用固定命令汇总最近验证结果
- 用更自动的方式触发分线程工作

## Risks

### Risk 1: 驾驶舱变成长文档

如果每次都往里堆历史，驾驶舱会失去“重新进入项目入口”的作用。

缓解方式：

- 限制栏目条数
- 只保留近期高信号内容
- 更老的历史继续放回 specs / plans / guide 文档

### Risk 2: 分线程目标不清

如果分线程拿到的不是一个单一任务，而是一个模糊方向，它就会重新变成另一个主线程。

缓解方式：

- 主线程必须先给出一句明确任务描述
- 一个分线程只做一件事

### Risk 3: 收尾时不更新驾驶舱

如果执行做完但没有回写，长期记忆仍然留在聊天里，系统就会退化回原状。

缓解方式：

- 把更新驾驶舱当作任务完成的一部分
- 没有更新驾驶舱，就不算真正完成收尾

## Success Criteria

这套设计成功的标志不是“线程很多”或“自动化很多”，而是下面三点：

- 维护者离开项目几天后，仍能在 2 分钟内知道当前状态
- 维护者每次都能看到唯一推荐的下一步
- 分线程完成后，长期有价值的信息能稳定回写到文档，而不是散落在历史对话里

## Summary

这次设计的本质是把项目从“靠记忆继续”切换成“靠驾驶舱继续”。

具体工作方式是：

- 用 `docs/current-state.md` 作为项目首页
- 用 1 个主线程负责判断主线
- 用少量短生命周期分线程负责执行
- 用固定回写流程把结果沉淀回文档

这样项目不会因为线程变长而失控，也不会因为线程过多而失去主线。
