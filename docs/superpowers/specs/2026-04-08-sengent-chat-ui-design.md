# Sengent Chat UI Design

## Goal

为 `sentieon_assist chat` 增加一层非全屏的终端交互壳层，让它在保留现有命令行工作方式的前提下，具备更接近 Claude Code 的启动欢迎区、消息排版和事件流体验。

第一版品牌名统一使用 `Sengent`，界面整体使用中文。

## Why

当前 `chat` 模式已经具备基础能力：

- 逐轮问答
- 缺失信息补问
- 本地资料检索
- 最终答案流式输出

但它的界面仍然停留在最原始的 `input()/print()` 形态，存在几个明显问题：

- 启动后没有引导区，新用户不知道可以怎么提问
- 用户输入、处理中间态和最终答案混在一起，层次不清
- 当前只有一条 `思考中...` 动画，无法体现真实处理阶段
- 现有体验更像脚本，不像一个可长期使用的命令行助手

用户已经确认，这次改版的目标不是全屏 TUI，而是先做一个更强的命令行壳层。

## Scope

第一版覆盖：

- 每次启动 `chat` 时固定显示一次 `Sengent` 欢迎面板
- 欢迎面板偏引导，不偏运行状态
- 聊天界面使用中文标签和中文提示
- 每轮对话显示清晰的用户消息区、事件流区和最终回答区
- 事件流只展示真实发生的内部阶段
- 使用 `rich` 构建非全屏终端渲染
- 保持现有 `chat` 命令和普通输入式交互方式
- 保持现有 `stream` 输出语义，但改为由 UI 层控制最终回答显示

第一版不做：

- 全屏 TUI
- 键盘快捷键系统
- 历史会话持久化
- Recent activity 面板
- 虚构工具调用轨迹
- 非 `chat` 子命令的富界面化

## UX Direction

整体风格参考 Claude Code 的终端气质，但不直接复制英文界面：

- 品牌：`Sengent`
- 语言：中文
- 形态：非全屏、自然向下滚动的命令行
- 视觉：细边框、轻量标题、低饱和强调色、清晰留白
- 节奏：先显示过程，再显示最终回答

核心原则：

- 不伪装成不存在的 agent 能力
- 不引入重绘复杂度过高的终端机制
- 不破坏现有命令行可用性
- 让“过程感”来自真实阶段，而不是 UI 装饰

## Welcome Panel

欢迎面板每次启动都显示，位于聊天循环开始之前。

面板结构：

- 顶部：`Sengent` 名称和简短版本标识
- 左侧：欢迎语、产品定位、2-3 条示例提问
- 右侧：`我可以帮你做什么`、`常用命令`、`提问建议`
- 底部：细分隔线，下面才进入正常聊天输入

欢迎面板偏引导：

- 示例提问要贴近当前真实能力，例如许可证报错、安装问题、模块查询
- 常用命令只包含当前已有命令，例如 `/quit`、`/reset`
- 不展示伪造的“最近活动”
- 不强调模型、资料目录、版本探针等运行信息

## Conversation Layout

每轮对话固定按以下顺序输出：

1. `你`
2. `事件流`
3. `Sengent`

### User Message

用户输入显示为一条简洁消息头：

- 标签为 `你`
- 保留原始输入文本
- 不需要做成大型卡片

### Event Stream

事件流用于表达真实内部阶段，形式是轻量垂直日志，不是工具面板。

第一版建议覆盖这些事件类型：

- `已识别问题类型`
- `正在检查缺失信息`
- `发现需要补充的信息`
- `正在检索本地资料`
- `正在整理参考答案`
- `正在生成回复`

事件流规则：

- 只显示真实发生的阶段
- 同一轮通常控制在 2-4 条
- 如果直接命中确定性回答，也要体现“资料命中”而不是“模型生成”
- 不能伪造 `Search`、`Write`、`Edit` 这类当前不存在的工具轨迹

### Final Answer

最终回答单独成块显示：

- 标题使用 `Sengent`
- 正文保留现有答案结构，例如 `【问题判断】`、`【模块介绍】`、`【常用参数】`
- 如果是流式输出，只对最终回答正文做流式显示
- 事件流必须先结束，再开始流式正文

## Technical Design

### Rendering Layer

新增一个独立的 chat UI 渲染层，使用 `rich` 而不是手写 ANSI。

选择 `rich` 的原因：

- 可以在非全屏模式下稳定渲染 panel、rule、颜色和换行
- 更适合欢迎区和消息块布局
- 可以降低手工处理终端宽度、颜色和边框的复杂度
- 不需要引入完整 TUI 框架

### Module Boundaries

建议新增：

- `src/sentieon_assist/chat_ui.py`
- `src/sentieon_assist/chat_events.py`

职责划分：

- `chat_ui.py`
  - 构建 `rich.console.Console`
  - 渲染欢迎面板
  - 渲染用户消息
  - 渲染事件流
  - 渲染最终回答
  - 处理流式回答显示
- `chat_events.py`
  - 定义事件类型
  - 提供将当前聊天路径映射成事件文本的帮助函数
- `cli.py`
  - 保留命令入口和聊天主循环
  - 在关键步骤前后发出事件
  - 把原先直接 `print` 的职责移交给 UI 层

## Runtime Behavior

`chat` 模式行为调整为：

1. 预热模型
2. 输出 `Sengent` 欢迎面板
3. 进入普通输入循环
4. 每轮输出用户消息块
5. 输出事件流
6. 输出最终回答块

业务语义保持不变：

- `/quit` 仍然退出
- `/reset` 仍然清空补问上下文
- `run_query()` 的判断逻辑不因 UI 改版而改变
- `doctor`、`sources`、`search`、单次 query 模式保持纯文本输出

## Streaming Rules

第一版流式策略：

- 事件流永远先输出完整
- 最终回答才进行流式写入
- 如果回答本身是结构化稳定文本，可直接静态显示
- 如果是模型润色输出，则在 `Sengent` 回答区中进行流式展开

这样可以避免：

- 状态和正文交错闪烁
- 光标定位混乱
- 事件和正文混成同一段输出

## Testing

新增和调整测试应覆盖：

- 每次进入 `chat` 都显示欢迎面板
- 聊天顺序为：欢迎面板 → 用户输入 → 事件流 → 最终回答
- 缺失信息路径会输出对应事件
- 资料查询路径会输出对应事件
- 直接确定性回答和模型生成回答都能正确显示
- `/quit`、`/reset` 行为不变
- 非 `chat` 模式输出不被富界面污染

建议新增：

- `tests/test_chat_ui.py`

并更新：

- `tests/test_cli.py`

## Risks

主要风险：

- `rich` 引入后，现有测试需要从纯字符串断言调整为更高层输出断言
- 流式输出和普通 `print` 的职责切分如果处理不好，容易产生重复换行或顺序错乱
- 如果 UI 和业务逻辑耦合过深，后续想升级到全屏 TUI 会很难迁移

控制方式：

- UI 层只管渲染，不承载业务判断
- 事件类型保持有限且可预测
- 先以非全屏模式做稳定输出，再考虑未来升级
